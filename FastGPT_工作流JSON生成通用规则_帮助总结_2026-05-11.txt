# FastGPT 工作流 JSON 生成通用规则文件带来的帮助总结

文件名：`FastGPT_工作流JSON生成通用规则_给AI模型使用_2026-05-11.md`

---

## 一、文件的核心价值

这个文件最大的帮助是：

> 把 FastGPT 工作流 JSON 的生成方式，从“AI 自由发挥、凭感觉编写”，变成了“按规则、按模板、按白名单节点、按自检流程生成”。

它不是普通说明文，而是一个面向 AI 模型的工作流 JSON 生成规范。  
其他 AI 模型看完后，可以知道：

- 哪些节点能用；
- 哪些字段不能乱写；
- 变量怎么引用；
- 节点怎么连接；
- 生成后怎么检查；
- 哪些节点应该谨慎使用；
- 哪些信息不能写死；
- 最终输出应该由哪个节点负责。

---

## 二、带来的主要帮助

### 1. 降低 AI 乱生成 JSON 的概率

以前让 AI 直接生成 FastGPT 工作流 JSON，很容易出现：

- 节点类型写错；
- 字段缺失；
- 连线 `source / target` 不存在；
- 变量引用错误；
- 输出字段没有声明；
- 私有 ID 被写死；
- AI 节点输出混乱；
- 指定回复节点无法正确引用上游变量；
- 看起来像 JSON，但导入后不能用。

这个文件通过规则限制，减少了这些问题。

---

### 2. 明确了 FastGPT JSON 的基本结构

文件规定顶层结构只保留：

```json
{
  "nodes": [],
  "edges": [],
  "chatConfig": {}
}
```

这能防止 AI 乱加环境相关字段，例如：

- `_id`
- `appId`
- `userId`
- `teamId`
- `datasetId`
- `toolId`
- `chatConfig._id`

这些字段很多都和个人账号、具体应用、数据库、知识库、工具配置有关，不能随便写死。

---

### 3. 建立了节点白名单

文件明确列出哪些节点适合自动生成或半自动生成，例如：

- 系统配置；
- 流程开始；
- 指定回复；
- 代码运行；
- 判断器；
- AI 对话；
- 文档解析；
- 问题分类；
- 知识库搜索；
- 文本拼接；
- 文本内容提取。

这让 AI 不会随便编造不存在或不稳定的节点。

---

### 4. 建立了高风险节点黑名单

文件也说明了一些暂不推荐自动生成的节点，例如：

- 用户选择；
- 表单输入；
- 工具调用；
- 变量更新；
- 批量执行。

这些节点结构复杂、依赖环境或交互状态，直接自动生成容易失败。

这能提醒 AI：

> 不知道真实导出结构时，不要猜。

---

### 5. 规范了变量引用方式

FastGPT 的变量引用很容易出错。  
这个文件明确规定变量引用使用：

```json
["节点ID", "输出ID"]
```

并且要求：

- 节点 ID 必须存在；
- 输出 ID 必须在该节点 `outputs` 中存在；
- 代码节点 `return` 的字段必须在 `outputs` 中声明；
- 后续节点只能引用真实存在的上游输出。

这对保证节点之间能正确传值非常重要。

---

### 6. 规范了指定回复节点的写法

文件区分了两种情况。

固定文本回复：

```json
"connected": false,
"selectedTypeIndex": 0
```

变量引用回复：

```json
"connected": true,
"selectedTypeIndex": 1
```

这个细节非常关键。  
如果混用，FastGPT 可能会把变量引用当成普通文本，导致最终回复不能正常显示上游结果。

---

### 7. 提供了节点 JSON 母版

文件中包含多个可复用节点母版，例如：

- `userGuide` 母版；
- `workflowStart` 母版；
- `answerNode` 母版；
- `code` 母版；
- `chatNode` 母版；
- `ifElseNode` 母版；
- `classifyQuestion` 母版；
- `datasetSearchNode` 母版。

这些母版能让 AI 不再从零编写节点结构，而是替换少量参数生成节点。

这相当于给 AI 提供了一套“积木库”。

---

### 8. 提高工作流生成的可复用性

有了这个文件后，生成新工作流时可以按固定思路：

```text
分析用户需求
↓
选择合适模板
↓
组合节点母版
↓
替换节点名称、提示词、变量引用
↓
生成 nodes / edges / chatConfig
↓
执行自检
↓
输出 JSON
```

这比每次重新摸索要稳定很多。

---

### 9. 帮助其他 AI 模型理解 FastGPT 工作流

这个文件的一个重要作用是“给其他 AI 模型使用”。

它把很多人工经验写成明确规则，包括：

- 哪些字段不能写；
- 哪些节点能用；
- 哪些节点慎用；
- 变量怎么传；
- 回复怎么输出；
- AI 节点怎么关闭中间输出；
- 代码节点怎么声明输出；
- 判断器分支怎么连；
- 问题分类分支怎么连；
- 生成后怎么检查。

这样其他 AI 不需要完全懂 FastGPT 源码，也能按规则生成更接近可用的工作流。

## 四、它对生成工作流的实际帮助

### 1. 让生成过程更有边界

没有这个文件时，AI 可能会为了满足用户需求而随意创造节点、字段或结构。

有了这个文件后，AI 必须先看：

- 这个节点是否在白名单里；
- 是否有母版；
- 是否需要人工配置；
- 是否存在私有 ID；
- 变量是否能引用成功；
- 输出字段是否声明；
- 连线是否真实存在。

这让生成过程更可控。

---

### 2. 让工作流更容易导入

FastGPT 的 JSON 导入并不是只要 JSON 格式正确就能成功。

还要保证：

- 节点字段符合平台结构；
- 连线 handle 大致正确；
- 节点类型存在；
- 输出字段存在；
- 引用路径能解析；
- 配置项不包含错误私有 ID。

这个文件能显著提高导入成功率。

---

### 3. 让后续修改更方便

当工作流出问题时，可以按照文件中的规则反查：

- 是节点问题；
- 是连线问题；
- 是变量引用问题；
- 是 outputs 未声明；
- 是指定回复节点写法错误；
- 是 AI 节点直接输出问题；
- 是字段写死问题；
- 是版本兼容问题。

这样排错速度会更快。

---

### 4. 让工作流可以被复制和扩展

因为文件中保存的是通用规则和节点母版，所以后续可以基于它生成不同类型的工作流，例如：

- 翻译助手；
- 票据识别助手；
- 会议纪要助手；
- 数据清洗助手；
- 文档总结助手；
- 表格分析助手；
- 问答分类助手；

- 报告生成助手；
- 知识库问答助手。

同一套规则可以反复复用。
