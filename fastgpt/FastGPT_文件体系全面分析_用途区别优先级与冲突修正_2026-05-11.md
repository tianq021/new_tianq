## 1. 总结论

你现在这些文件不是乱的，而是分成了 5 类：

```text
1. 实战测试记录类
2. 通用规则类
3. 规则价值说明类
4. 源码约束修订类
5. 单节点母版与拼接规则类
```

最重要的结论是：

> 真正给 AI 生成 FastGPT 工作流 JSON 时，不能只看一个文件。  
> 最稳的组合是：  
> **源码约束修订版 + 单节点母版与拼接规则 + 目标与模板追加记录。**

简单理解：

```text
源码约束修订版：
告诉 AI 不能乱写什么，哪些要手动配置。

单节点母版与拼接规则：
告诉 AI 每个节点具体怎么写、怎么连。

目标与模板追加：
告诉 AI 以前踩过哪些坑，哪些模板已经实测成功。

帮助总结：
告诉人或 AI 这个规则体系有什么价值，偏说明和报告。

旧版给AI模型使用：
早期规则文件，有价值，但已被源码约束修订版和单节点母版拆分、修正、增强。
```

### 3.1 `FastGPT_一句话生成工作流JSON_目标与模板追加_2026-05-10.txt`

#### 文件定位

这是 **实战测试记录 + 纠错增补 + 已验证模板记录**。

它不是干净的规则手册，而是把整个实验过程、错误、修正、成功结果全部记下来了。

#### 里面主要有什么

这个文件主要记录：

1. 一句话生成 FastGPT 工作流 JSON 的真正目标；
2. AI 节点 401 的历史问题；
3. 后续云版 4.15.0 上 AI 节点可用的更新；
4. `simple_reply` 模板成功；
5. `ai_chat` 模板成功；
6. `code_task.no_input` 随机数模板成功；
7. `answerNode` 引用代码节点输出时必须引用 `output.id`，不能只看 `output.key`；
8. `readFiles` 输出应优先使用 `system_text / system_rawResponse / system_error_text`；
9. 文件上传引用应优先注意 `userFiles` 的输出声明和引用格式；
10. 最终路线应是：AI 输出 plan，代码模板库生成真实 JSON，而不是让 AI 直接自由生成完整 `nodes / edges`；
11. 知识库模板卡片方案；
12. 模板库扩展路线。

#### 它的核心价值

这个文件最重要的不是“提供节点模板”，而是：

```text
防止下一个 AI 或下一个对话重复犯旧错误。
```

例如它明确记录：

```text
AI 不应该直接生成完整 nodes/edges。
AI 应该输出 plan。
代码节点根据白名单 template_id 拼装真实 JSON。
```

还记录了随机数模板的关键坑：

```text
answerNode 不能引用 ["code_task_node", "result"]
而应该引用代码节点 outputs 里的 id，例如：
["code_task_node", "code_result_output"]
```

#### 适合用途

- 做项目历史记录；
- 做纠错资料；
- 给 AI 解释为什么采用 plan + 模板库；
- 建立错误案例库；
- 判断某个模板是否已经实测成功；
- 规划后续模板数量。

#### 不适合用途

- 不适合直接当“最终生成规则”；
- 不适合让 AI 从里面复制早期错误尝试；
- 不适合不加筛选地放进提示词最前面。

#### 使用优先级

```text
高。
```

但它的角色是 **历史纠错依据**，不是最终模板执行手册。

### 3.4 `FastGPT_工作流JSON生成通用规则_给AI模型使用_2026-05-11.md`

#### 文件定位

这是 **旧版给 AI 模型使用的通用规则文件**。

它比“帮助总结”更具体，因为它不是只解释价值，而是真的写了：

- 顶层结构；
- 节点白名单；
- 节点母版；
- 变量引用；
- 工作流组合模板；
- 自检清单；
- 常见错误；
- 给其他 AI 的强制指令。

#### 里面主要有什么

包括：

1. 顶层结构只能有 `nodes / edges / chatConfig`；
2. 不要写死 `_id / appId / userId / teamId / datasetId / toolId / chatConfig._id`；
3. 已验证节点白名单；
4. 高风险节点；
5. `userGuide` 母版；
6. `workflowStart` 母版；
7. `answerNode` 固定文本和变量引用母版；
8. `code` 母版；
9. `chatNode` 母版；
10. `ifElseNode` 母版；
11. `classifyQuestion` 母版；
12. `datasetSearchNode` 母版；
13. `readFiles` 母版；
14. `httpRequest468` 母版；
15. `contentExtract` 母版；
16. `textEditor` 母版；
17. 组合模板库；
18. 用户需求到模板的选择规则；
19. 自检清单；
20. 最小固定回复完整 JSON 示例。

#### 它的价值

它是早期很重要的“总文件”，因为它试图让一个 AI 模型看完后就能生成工作流。

#### 但是它现在的问题

这个文件已经被后来的两个文件进一步拆分和修正：

```text
源码约束修订版
单节点母版与拼接规则
```

所以旧版有些地方可能不再是最高优先级。

#### 需要特别注意的潜在冲突

##### 冲突 1：readFiles 文件引用格式

旧版文件中有地方写：

```json
"value": ["448745", "userFiles"]
```

但后续纠错记录里多次强调，文件上传类 `fileUrlList` 更应该优先参考旧式嵌套格式：

```json
"value": [["448745", "userFiles"]]
```

因此，如果生成文件上传、文档解析、图片识别类工作流，应以“目标与模板追加”和“源码约束修订版/单节点母版”的最新规则为准。

##### 冲突 2：代码节点输出 id

旧版可能把代码节点输出写成：

```json
"id": "result",
"key": "result"
```

这在简单情况下可能能用，但后续随机数实测发现，云版 4.15.0 更稳定做法是：

```json
{
  "id": "code_result_output",
  "key": "result",
  "valueType": "string",
  "label": "result"
}
```

然后 `answerNode` 引用：

```json
["code_task_node", "code_result_output"]
```

也就是引用 `output.id`，不是只按 `key=result`。

##### 冲突 3：AI 是否直接生成完整 JSON

旧版虽然强调“不能自由编造”，但整体还是偏向“AI 可以根据规则生成 nodes/edges”。

后续实测路线更稳：

```text
AI 只输出 plan
代码模板库生成真实 JSON
```

因此新路线优先级更高。

#### 使用建议

保留这个文件，但标记为：

```text
旧版通用规则 / 历史版 / 可参考但不是最高优先级
```

如果它和“源码约束修订版”冲突：

```text
以源码约束修订版为准。
```

如果它和“目标与模板追加”里的实测结论冲突：

```text
以实测结论为准。
```

#### 使用优先级

```text
中。
```

### 3.5 `FastGPT_工作流JSON生成通用规则_给AI模型使用_2026-05-11_源码约束修订版.md`

#### 文件定位

这是 **当前最高规则文件**。

它的作用是从源码和真实测试角度修正旧版。

#### 它主要解决什么

它解决旧版容易让 AI 误解的问题：

1. AI 不能凭空创造 FastGPT 节点；
2. 节点必须来自源码真实存在的 `flowNodeType`；
3. 简单工作流可以生成完整 JSON；
4. 中等工作流生成 JSON 模板并提醒手动配置；
5. 复杂工作流只能生成骨架或设计方案；
6. 模型、知识库、API、数据库、toolId 等必须手动配置；
7. 文件引用、判断器分支、分类分支、readFiles 输出等必须按真实规则；
8. `selectedTypeIndex` 比旧的 `connected` 更关键；
9. 生成后必须自检；
10. 不确定就提醒人工配置，不允许编造。

#### 它和旧版的区别

| 对比项 | 旧版给AI模型使用 | 源码约束修订版 |
|---|---|---|
| 重点 | 让 AI 会生成 JSON | 限制 AI 不乱生成 |
| 风格 | 总纲 + 母版 + 示例 | 源码事实 + 禁止项 + 手动配置 |
| 对复杂流程 | 容易想尝试生成 | 明确分 A/B/C 类 |
| 对私有配置 | 有提醒 | 更强硬，不能编造 |
| 对节点 | 白名单 | 白名单 + 源码约束 + 稳定等级 |
| 对输出 | 生成 JSON | 生成 JSON / 模板 / 设计方案三类 |

#### 适合用途

- 给 AI 的最高系统规则；
- 判断一个需求能不能自动生成；
- 判断哪些地方必须手动配置；
- 排除旧文件冲突；
- 作为知识库最高优先级文档。

#### 不适合用途

- 它不是详细节点母版库；
- 它需要配合“单节点母版与拼接规则”使用。

#### 使用优先级

```text
最高。
```

这几个文件的关系可以理解成：

```text

    ↓

    ↓

单节点母版与拼接规则
    ↓

```

最终给 AI 使用时，最合理的读取顺序是：

```text
1. 源码约束修订版
2. 单节点母版与拼接规则
3. 目标与模板追加
4. 旧版给AI模型使用
```

## 6. 哪些文件可以合并，哪些不要合并

### 6.1 不建议合并的文件

以下两个不要合并成一个大文件：

```text
源码约束修订版.md
单节点母版与拼接规则.md
```

原因：

```text
一个管规则边界，一个管节点写法。
合并后太大，AI 容易找不到重点。
```

### 6.2 可以合并或转化的文件

`帮助总结.txt` 和 `帮助总结.md` 可以保留一个正式版：

```text
建议保留：帮助总结.md
TXT 作为备份归档。
```

`旧版给AI模型使用.md` 可以归档：

```text
不要删除，但放 archive 或标注“旧版”。
```

### 6.3 `目标与模板追加` 不建议删除

虽然它很长，但里面有大量实测记录和纠错信息。

建议保留为：

```text
实验记录 / 测试日志 / 错误案例库
```

## 10. 冲突修正规则

当文件之间结论不一致时，按这个优先级判断：

```text
实测成功导出结构
>
目标与模板追加里的后续修正
>
源码约束修订版
>
单节点母版与拼接规则
>
旧版给AI模型使用
>
帮助总结
```

### 10.1 关于 `answerNode`

最终规则：

```text
answerNode 输入 key 必须是 text。
固定文本：selectedTypeIndex=0。
变量引用：selectedTypeIndex=1。
引用代码节点输出时，第二项应引用 outputs.id。
```

### 10.2 关于代码节点输出

最终规则：

```text
function main(params) 必须 return object。
return { result: "..." } 时，outputs 里必须有 key=result。
answerNode 引用时更稳的是引用该输出项的 id，例如 code_result_output。
```

### 10.3 关于 readFiles

最终规则：

```text
readFiles 输出文本优先用 system_text。
不要写 text/rawResponse/error。
使用 userFiles 时，workflowStart.outputs 必须有 userFiles。
chatConfig.fileSelectConfig 必须开启。
```

### 10.4 关于文件上传引用

最终规则：

```text
文件列表类 fileUrlList 优先参考真实导出结构和最新修订版。
如果旧版写 ["448745", "userFiles"]，而修订版/实测记录写 [["448745", "userFiles"]]，优先用后者或以当前平台导出母版为准。
```

### 10.5 关于 AI 生成完整 JSON

最终规则：

```text
不要让 AI 自由生成完整 nodes/edges。
更稳路线是：
AI 输出 plan
代码模板库根据白名单 template_id 生成真实 JSON。
```

### 10.6 关于知识库、工具、数据库、HTTP

最终规则：

```text
datasetId、toolId、API Key、数据库连接、webhook、私有模型 ID 一律不能编造。
只能提醒导入后手动配置，或者生成半自动模板。
```

## 13. 最终建议

你现在不需要继续堆很多重复总结文件。

下一步最应该做的是：

```text
1. 保留 P0 文件：
   - 源码约束修订版
   - 单节点母版与拼接规则

2. 把目标与模板追加作为错误案例库和测试日志保留。

3. 把帮助总结作为说明文档保留。

4. 把旧版给AI模型使用移动到 archive，避免和修订版混用。

5. 后续每新增一个成功模板：
   - 先手动搭建
   - 导出 JSON
   - 提取单节点母版
   - 写入单节点母版文档
   - 在目标与模板追加里记录成功/失败
   - 在源码约束修订版里更新白名单或手动配置提醒
```

一句话：

> 你的 FastGPT 文档体系已经从“聊天记录”升级成了“工作流生成知识库雏形”。  
> 现在最重要的是控制优先级，不要让旧版规则和新版修正规则混在一起误导 AI。
