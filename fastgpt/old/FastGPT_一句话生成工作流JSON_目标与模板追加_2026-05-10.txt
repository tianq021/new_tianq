# FastGPT 工作流 JSON 生成与测试全过程总结【纠错增补版】

记录时间：2026-05-10
修订目的：把旧总结中容易误导下一个对话的内容全部修正，尤其是 AI 节点、模型 401、readFiles 变量格式、旧导出版结构、不能写死的字段、当前真正目标。

重要提醒：
本文件应优先于旧版 `FastGPT_workflow_JSON_experiment_summary_2026-05-10.txt` 使用。旧版里面有些结论是当时阶段性推测，现在已经被后续测试修正。下一个对话不要只看旧版，否则容易继续往错误方向生成 JSON。

======================================================================
一、最终目标【已修正】
======================================================================

真正目标不是单纯验证 readFiles、CSV 直读、文件后缀分流，也不是做一个固定小 demo。

真正目标是：

> 做一个 FastGPT 工作流生成器：用户只输入一句话，FastGPT 内部的 AI 节点理解需求，然后生成可导入的 FastGPT 工作流 JSON，从而减少重复搭建工作流的麻烦。

目标链路应该是：

用户一句话需求
↓
AI 对话节点理解需求
↓
AI 生成 FastGPT 工作流 JSON
↓
代码节点清洗/校验 JSON
↓
指定回复输出 JSON
↓
用户复制/导入生成的新工作流

因此：
1. 这个最终目标必须包含 AI 节点。
2. 没有 AI 节点，只靠代码节点，只能生成几个写死模板，不能真正理解“一句话需求”。
3. v19 / v20 的非 AI 版本只能证明“代码节点可运行、指定回复可输出、JSON 结构可导入”，不能代表最终目标完成。
4. 如果 FastGPT 平台 AI 节点模型调用本身 401，那么“一句话生成工作流 JSON”的 FastGPT 内部版就暂时无法实现。
5. 这种情况下，可以退而求其次：在 ChatGPT 对话里生成 FastGPT JSON 文件，再手动导入 FastGPT，但这已经不是“FastGPT 内部一句话生成工作流”。

======================================================================
二、当前最关键结论【防误导版】
======================================================================

1. 当前最关键问题不是 JSON 结构，而是 FastGPT 平台模型调用报：

401 Invalid token

2. 这个错误已经在“手动新建普通 AI 应用”里出现，不是只发生在导入 JSON 的工作流里。

截图/测试现象：
- 手动新建应用。
- AI 配置里选择 qwen-plus。
- 提示词只写“你只需要回复就行”。
- 输入“你好”。
- 仍然弹出 401 Invalid token。

结论：

> 至少 qwen-plus 这个模型渠道/API Key/Token/Base URL 配置有问题。

3. 不能直接说“所有 AI 节点都不能用”，除非继续测试：
- 手动新建 AI 应用，换 qwen-vl-max 是否能跑；
- 手动新建 AI 应用，换其他模型是否能跑；
- 老工作流里的 AI 节点是否也 401；
- 其他账号/其他同学是否也 401。

4. 目前更准确说法：

> qwen-plus 已确认不可用或配置异常；qwen-vl-max 是否可用需要单独手动测试；如果所有模型手动测试都 401，才说明整个平台 AI 模型调用配置整体异常。

5. 如果手动新建 AI 节点都 401，那么后面无论 JSON 写得多正确，只要包含 AI 节点都会失败。

======================================================================
三、旧总结里需要纠正的误导点
======================================================================

【误导点 1】把 readFiles 当成当前主线。

旧总结中 v10/v11 大量围绕 readFiles 输出空继续排查，这在当时没错，但现在主线已经变了。

修正：
当前主线不是“让 CSV/readFiles 成功解析”，而是“做一句话生成工作流 JSON 的 AI 生成器”。readFiles 只是未来生成文件处理类工作流时可能用到的节点，不是当前最核心问题。

----------------------------------------------------------------------

【误导点 2】把 v11 的 readFiles 直接引用当成正确方向。

旧版写过：

value: ["workflowStartNodeId", "userFiles"]

并把它作为 v11 尝试方向。

修正：
从用户自己导出的多个真实工作流看，readFiles 和 AI 节点接收 userFiles 时更稳定、更应该优先参考旧式嵌套格式：

"value": [
  [
    "448745",
    "userFiles"
  ]
]

真实来源：
- 票据识别小助手V3.json 的 readFiles 使用这个格式。
- 数据清理小助手_去重导出.json 的 readFiles 使用这个格式。
- 票据识别小助手V3.json 的图片 AI 节点 fileUrlList 也使用这个格式。
- 1 (1).json 的 qwen-vl-max AI 节点 fileUrlList 也使用这个格式。

因此后面生成旧版 FastGPT JSON 时，文件上传引用优先用：

[["448745", "userFiles"]]

不要优先用：

["448745", "userFiles"]

除非是明确测试直接引用格式。

----------------------------------------------------------------------

【误导点 3】readFiles 输出 key 写错/不统一。

旧版 v10/v11 中曾写：

outputs: text / rawResponse / error

修正：
用户真实导出的 readFiles 节点输出是：

system_text
system_rawResponse
system_error_text

例如真实结构：

{
  "id": "system_text",
  "key": "system_text",
  "valueType": "string",
  "type": "static"
}

{
  "id": "system_rawResponse",
  "key": "system_rawResponse",
  "valueType": "arrayObject",
  "type": "static"
}

{
  "id": "system_error_text",
  "key": "system_error_text",
  "valueType": "string",
  "type": "error"
}

所以下游引用 readFiles 文本时应使用：

{{$readFilesNodeId.system_text$}}

或者输入 value 引用：

["readFilesNodeId", "system_text"]

不要再默认写 text/rawResponse/error。

----------------------------------------------------------------------

【误导点 4】以为 AI 节点结构完全错误。

修正：
后续上传的 1.json、1 (1).json、2.json 已经证明 AI 节点结构是存在且可导入的，真实 AI 节点类型是：

flowNodeType: "chatNode"
version: "4.9.7"
avatar: "core/workflow/template/aiChat"

AI 节点报 401，不等于 AI 节点 JSON 结构错。401 是模型授权/Token/API Key/Base URL/渠道配置问题。

----------------------------------------------------------------------

【误导点 5】以为 v18 不写模型就能绕过 401。

修正：
即使不写或清空 model，FastGPT 仍可能使用默认模型/残留模型/平台默认配置，仍可能触发 401。

所以“模型名不写死”不等于“不调用 token”。

AI 节点只要运行，就一定会调用模型渠道；模型渠道配置坏了就会 401。

----------------------------------------------------------------------

【误导点 6】以为 qwen-vl-max 一定能用。

修正：
qwen-plus 已经手动测试 401。
qwen-vl-max 只是因为某些导出文件里存在，且没有先经过 qwen-plus，所以“可能能用”。
但必须手动新建 AI 应用选择 qwen-vl-max 测试后才能确认。

不能在总结里写死“qwen-vl-max 能用”。
只能写：

> qwen-vl-max 可能能用，但未最终确认；需要手动测试。

----------------------------------------------------------------------

【误导点 7】把 2.json 直接当可用母版。

修正：
2.json 的 AI 节点都是 qwen-vl-max，表面看更适合当 AI 母版。
但它有一个问题：AI 节点里 fileUrlList 引用了 userFiles：

[["448745", "userFiles"]]

可是 2.json 的 workflowStart.outputs 里只有 userChatInput，没有 userFiles。

这说明 2.json 不能直接照抄，需要补 workflowStart 的 userFiles 输出，或者如果不测文件上传，就把 AI 节点的 fileUrlList 留空。

----------------------------------------------------------------------

【误导点 8】把 answerNode 的变量引用只理解成 value 数组。

修正：
指定回复 answerNode 有两种常见写法。

方式 A：整个输入引用某个上游输出：

"value": ["text_file_result", "system_text"]

方式 B：在文本内容里插变量：

"value": "结果：{{$codeNodeId.result$}}"

用户真实导出的工作流里，大量使用方式 B：

{{$r2WolLufOUO6oLwc.qLUQfhG0ILRX$}}
{{$gX18NQYSYef5t7yZ.result$}}

因此后面不能只使用一种写法。简单输出上游完整结果时可用方式 A；需要混排 Markdown/文字/多个变量时用方式 B。

----------------------------------------------------------------------

【误导点 9】把 chatConfig.fileSelectConfig 当成总是必须。

修正：
如果工作流需要上传文件，chatConfig 里最好有 fileSelectConfig，并且 workflowStart.outputs 必须有 userFiles。

如果只是纯文本 AI 对话，不一定必须有 fileSelectConfig。

但如果 AI 节点或 readFiles 引用了 userFiles，而 workflowStart 没有 userFiles 输出，那就是错误引用。

----------------------------------------------------------------------

【误导点 10】把 v15/v16/v17/v18/v19/v20 都当作有效推进。

修正：
这些版本有些只是排查实验，不应该被下个对话当成正确最终方向。

- v16/v17/v18：包含 AI 生成 JSON 思路，但因为模型 401，无法验证真正运行。
- v19/v20：没有 AI 节点，只能生成写死模板，不能实现最终目标。
- v13/v14/v15：试图绕开 readFiles 直读 CSV，方向后来确认偏离最终目标。

真正可靠母版仍然是用户自己导出的旧工作流结构，尤其：
- 票据识别小助手V3.json：分支、AI、readFiles、代码、指定回复、数据库流程参考。
- 数据清理小助手_去重导出.json：readFiles + Python + 指定回复的干净参考。
- 薪资计算助手_加强版.json：一句话 → AI 提取 JSON → 代码处理 → 输出的逻辑参考。
- 2.json：qwen-vl-max AI 节点结构参考，但不能直接照抄 userFiles 引用。

======================================================================
四、已验证事实库【以后按这个判断】
======================================================================

1. 用户环境版本记录：

历史记录里有“学校部署的 FastGPT V4.14.6”。
因此后续生成 JSON 应优先参考用户自己导出的旧版结构，不要按 GitHub 仓库最新结构硬套。

2. 顶层结构：

用户真实导出的工作流顶层通常是：

{
  "nodes": [],
  "edges": [],
  "chatConfig": {}
}

不要乱加 metadata、schemaVersion 等未知字段。

3. 系统配置节点：

nodeId: "userGuide"
flowNodeType: "userGuide"
avatar: "core/workflow/template/systemConfig"
version: "481"

4. 流程开始节点：

nodeId 常见是 "448745"
flowNodeType: "workflowStart"
version: "481"

纯文本工作流至少有：
- userChatInput

文件上传工作流必须额外有：
- userFiles

5. 代码节点：

flowNodeType: "code"
avatar: "core/workflow/template/codeRun"
codeType: "js" 或 "py"

代码 return 的字段必须在 outputs 中声明，否则下游变量引用可能找不到或为空。

6. 指定回复节点：

flowNodeType: "answerNode"
avatar: "core/workflow/template/reply"
输入 key 必须是：text

不要用 answerText 当指定回复输入 key。

7. AI 节点：

flowNodeType: "chatNode"
avatar: "core/workflow/template/aiChat"
version: "4.9.7"

关键输入：
- model
- isResponseAnswerText
- systemPrompt
- history
- quoteQA
- fileUrlList
- userChatInput

关键输出：
- history
- answerText
- reasoningText
- system_error_text

8. AI 节点中间处理时：

如果 AI 节点只是用于生成 JSON 给后续代码清洗，不应该直接把 AI 回复输出给用户，应设置：

"isResponseAnswerText": false

如果只是最小 AI 对话测试，可设置：

"isResponseAnswerText": true

9. AI 节点引用文件：

优先参考用户真实导出格式：

"fileUrlList": {
  "value": [
    ["448745", "userFiles"]
  ]
}

如果不测试文件，fileUrlList 可为空数组：

"value": []

10. readFiles 节点：

flowNodeType: "readFiles"
version: "4.9.2"  是用户真实导出常见版本。
输入 key：fileUrlList
输入 value：[["448745", "userFiles"]]
输出 key：system_text / system_rawResponse / system_error_text

11. 判断器：

flowNodeType: "ifElseNode"
条件列表放在 inputs 的 ifElseList。
分支连线 sourceHandle 常见：
- nodeId-source-IF
- nodeId-source-ELSE

12. HTTP 节点：

用户真实导出里 HTTP 节点类型是：

flowNodeType: "httpRequest468"

不要随便写 httpRequest。

13. 数据库/工具节点：

用户票据流程里有 tool 节点用于调用数据库/存储过程。
这些节点高度依赖平台内部工具 ID、数据库连接、参数配置，不能在新 JSON 里凭空写死。

======================================================================
五、AI 节点 401 的正确判断流程
======================================================================

现在遇到 401 Invalid token 时，不要再优先怀疑 JSON 结构。

正确排查顺序：

1. 手动新建空应用。
2. 手动拖或使用默认 AI 对话节点。
3. 手动选择 qwen-plus。
4. 输入“你好”。
5. 如果 401，说明 qwen-plus 平台配置坏。
6. 手动换 qwen-vl-max。
7. 输入“你好”。
8. 如果 qwen-vl-max 能跑，则只避开 qwen-plus。
9. 如果 qwen-vl-max 也 401，再换其他模型。
10. 如果所有模型都 401，说明整个平台模型渠道/API Key/Base URL/Token/余额/权限大概率异常。
11. 如果手动能跑，但导入 JSON 不能跑，才回头检查 JSON 的 model 字段和节点结构。

401 常见原因：
- API Key 错误。
- API Key 过期。
- API Key 被删除。
- API Key 无权限访问该模型。
- Base URL 填错。
- 模型渠道配置错。
- 模型名和渠道不匹配。
- 账号欠费或额度用完。
- 平台共享 token 失效。
- 学校部署环境统一模型配置异常。

一句话：

> 401 是模型授权错误，不是工作流 JSON 语法错误。

======================================================================
六、不能写死清单【非常重要】
======================================================================

以下内容后续生成 JSON 时不能写死，或者必须明确说明“需要导入后手动改”：

1. 模型名

尤其不要写死：
- qwen-plus

因为 qwen-plus 已经在手动应用中触发 401。

qwen-vl-max 也不要绝对写死为“可用”，只能说“如果你平台能用 qwen-vl-max，可以暂时用它”。

2. API Key / Token

绝对不能写死。

3. Base URL

不同平台、学校部署、代理环境可能不同。

4. chatConfig._id

不要照抄旧文件里的 _id。
让 FastGPT 自动生成最安全。

5. datasetId / 知识库 ID

不同账号/应用不一样。

6. 工具 ID / tool ID / 数据库工具 ID

数据库、HTTP 工具、插件工具都不能凭空写死。

7. webhook 地址

例如飞书、企业微信、第三方接口 webhook，不能照抄。

8. 数据库连接、存储过程、表名、字段名

除非用户明确要继续使用已有的 MySQL 存储过程，例如 sp_save_receipt。

9. 第三方 API key

例如聚合数据天气/新闻 key 不能写进新测试文件。

10. 用户文件 URL

上传文件 URL 是临时链接，不能写死。

11. 自动生成的 nodeId 可复制但要保证唯一

nodeId 可以人工指定，但同一工作流内必须唯一。不要复制造成重复。

12. 变量引用不能指向不存在的输出

例如：
AI 节点 fileUrlList 引用了 ["448745", "userFiles"]，则 workflowStart.outputs 必须真的包含 userFiles。

13. readFiles 输出 key 不要自己改成 text/rawResponse/error

旧版真实输出应优先用 system_text / system_rawResponse / system_error_text。

======================================================================
七、当前推荐母版选择
======================================================================

按任务类型选择母版：

1. 最小文件上传/后缀识别母版

参考：v9 成功版思路。

用途：
- 验证 workflowStart.userFiles。
- 验证 code 节点。
- 验证 ifElseNode。
- 验证 answerNode。
- 不依赖 AI。

注意：
这是结构测试母版，不是最终 AI 生成器。

2. 文件解析 + Python 清洗母版

参考：数据清理小助手_去重导出.json

结构：

workflowStart(userFiles)
↓
readFiles
↓
code(py)
↓
answerNode

优点：结构清晰，节点少，适合参考 readFiles、Python、指定回复变量格式。

3. 图片/文档分流 + 票据流程母版

参考：票据识别小助手V3.json

用途：
- 复杂分支。
- 后缀判断。
- 图片走 AI 视觉。
- 文档走 readFiles。
- AI 输出 JSON。
- 代码校验。
- 数据库工具调用。

注意：复杂，不适合作为第一次测试生成器母版。

4. 一句话 → AI 结构化 JSON 母版

参考：薪资计算助手_加强版.json

用途：
- 用户自然语言输入。
- AI 提取结构化 JSON。
- 代码节点处理。
- 最终输出。

这和“一句话生成工作流 JSON”的逻辑最像。

5. 单 AI 节点母版

参考：1.json、1 (1).json、2.json

用途：
- 测 AI 节点是否能调用模型。
- 测模型 401 是否和 JSON 无关。

注意：
1.json 使用 qwen-plus，当前容易 401。
2.json 使用 qwen-vl-max，但 workflowStart 缺 userFiles，却被 AI 节点引用了，需要修。

======================================================================
八、当前建议的下一步测试顺序
======================================================================

不要继续一口气生成大流程。

正确顺序：

第一步：平台模型可用性测试

手动新建应用：

开始 → AI 对话

分别测试：
- qwen-plus
- qwen-vl-max
- 其他平台已有模型

目标：确认到底哪些模型可用。

第二步：单 AI 节点 JSON 测试

如果某个模型手动可用，再用导出的 AI 节点结构生成单 AI 节点 JSON。

结构：

workflowStart
↓
chatNode(isResponseAnswerText=true)

目标：确认导入 JSON 后 AI 节点也能跑。

第三步：AI 生成 JSON + 直接回复测试

结构：

workflowStart
↓
chatNode(isResponseAnswerText=true)

提示词让 AI 只输出一个极简 JSON，例如：

{"ok": true, "message": "hello"}

目标：确认 AI 能稳定输出 JSON 文本。

第四步：AI + 代码清洗 + 指定回复

结构：

workflowStart
↓
chatNode(isResponseAnswerText=false)
↓
code(js 清洗 JSON)
↓
answerNode(text 引用 code.result)

目标：这才是最终工作流生成器的核心结构。

第五步：再让 AI 生成 FastGPT 工作流 JSON

先限制只能生成最简单结构：

userGuide
workflowStart
code
answerNode
edges
chatConfig

不要一开始就让 AI 生成 readFiles、HTTP、数据库、知识库、工具调用。

第六步：逐步扩展节点库

按顺序加入：
- answerNode
- code
- ifElseNode
- textEditor
- readFiles
- chatNode
- httpRequest468
- datasetSearchNode
- tool

九、下一个对话必须知道的核心规则
======================================================================

1. 当前真正目标：做 FastGPT 内部“一句话生成工作流 JSON”的生成器。
2. 这个目标必须依赖 AI 节点。
3. 现在最大阻塞：FastGPT AI 模型调用 401 Invalid token。
4. 手动新建 qwen-plus 也 401，所以不是 JSON 导入问题。
5. 不能继续靠非 AI 代码模板冒充生成器。
6. 不能写死 qwen-plus。
7. qwen-vl-max 只能作为待测模型，不能断言一定可用。
8. 用户环境历史记录显示 FastGPT V4.14.6，应优先参考用户自己导出的旧 JSON。
9. 顶层结构只用 nodes / edges / chatConfig。
10. 系统配置节点是 userGuide，不是 systemConfig。
11. 指定回复 answerNode 输入 key 是 text，不是 answerText。
12. 代码节点 flowNodeType 是 code。
13. AI 节点 flowNodeType 是 chatNode。

======================================================================
十七、3C 直接让 AI 生成完整 JSON 的问题
======================================================================

3C 证明 AI 能输出 JSON，但也暴露了关键问题：

AI 直接生成完整 FastGPT 工作流 JSON 时，会自己猜结构。例如 3C 生成的 edges 是：

{
  "sourceNodeId": "448745",
  "targetNodeId": "code_node_1",
  "sourceHandleId": "userChatInput",
  "targetHandleId": "userChatInput"
}

但用户云版真实导出结构更接近：

{
  "source": "448745",
  "target": "answer_result",
  "sourceHandle": "448745-source-right",
  "targetHandle": "answer_result-target-left"
}

因此：

1. AI 直接生成 nodes/edges 不稳定。
2. AI 会使用 sourceNodeId/targetNodeId/sourceHandleId/targetHandleId 这种未验证字段。
3. 云版 4.15.0 可导入和可运行的边结构已确认优先使用 source/target/sourceHandle/targetHandle。
4. 所以不能让 AI 自由生成完整工作流 JSON。
5. 正确路线是：AI 只生成 plan，代码节点使用已验证模板拼 JSON。

最终固定原则：

> AI 不生成完整 nodes/edges。
> AI 只负责理解需求，输出 plan JSON。
> 代码节点根据 plan.workflow_type 调用模板函数生成工作流 JSON。

二十一、随机数 code_task 模板最终成功
======================================================================

最初生成随机数时报错 Data validation error，后续定位并修复。

随机数代码：

function main(params) {

  return { result: String(n) };
}

最初失败表现：
- 导入后节点出现。
- 代码节点里有代码。
- 指定回复引用“生成随机数 → result”。
- 运行时报 Data validation error。
- 代码节点输出区域只显示：
  - 完整响应数据 Object
  - 错误信息 String
- 没有显示 result String。

第一次错误判断：
以为代码节点 outputs 里添加了：
{
  id: "result",
  key: "result",
  valueType: "string"
}
就可以被引用。

后续真实导出成功工作流后发现：
云版 4.15.0 代码节点动态输出有一个独立随机 id，例如：

{
  "id": "qLUQfhG0ILRX",
  "type": "dynamic",
  "key": "result",
  "valueType": "string",
  "label": "result"
}

而 answerNode 引用的是：

"value": [
  "x2SKHIaS2Ra6SHGq",
  "qLUQfhG0ILRX"
]

也就是说：

> answerNode 引用的是 output.id，不是 output.key。

这就是最关键修正点。

----------------------------------------------------------------------

失败写法：

代码节点 nodeId:
code_task_node

代码节点输出：
{
  id: "result",
  key: "result"
}

answerNode 引用：
value: ["code_task_node", "result"]

这个可能导入后不稳定，云版没有正确注册自定义输出，或引用错误。

----------------------------------------------------------------------

成功写法：

代码节点 nodeId:
code_task_node

代码节点输出：
{
  id: "code_result_output",
  type: "dynamic",
  key: "result",
  valueType: "string",
  label: "result"
}

answerNode 引用：
value: ["code_task_node", "code_result_output"]

关键原则：

> answerNode 的 value 第二项必须引用目标输出的 id。
> key 只是变量名，给代码 return 对象对应使用；引用时不一定用 key。
> 自定义输出必须同时有 id 和 key。
> id 可以人工固定，只要同一工作流内唯一即可。
> 生成器里可以统一固定 id = "code_result_output"。

----------------------------------------------------------------------

  result String

  代码运行 → result

  480617
  370993
  后续又输出：
  449484
  388715
- 每次运行数字不同。

最终结论：

> code_task 随机数模板成功。
> 这是第三个真正可用模板。
> 之前 Data validation error 的根本原因不是代码本身，而是 answerNode 引用了 output key，而云版需要引用 output id。

二十六、知识库模板检索库设计
======================================================================

用户提出：
“做一个知识库？能不能让他从知识库搜索，需要哪个就返回哪个模板。”

结论：
可以，而且很适合下一阶段。但知识库不是直接存巨大 JSON 让 AI 粘贴，而是存“模板卡片”。

推荐架构：

流程开始
↓
AI：把用户一句话转成检索 query / 初步需求
↓
知识库搜索：从“FastGPT 工作流模板库”找匹配模板卡片
↓
AI：根据搜索结果选择 template_id，并补全 params
↓
代码节点：根据 template_id 调用固定模板函数
↓
指定回复：输出 JSON

知识库职责：
1. 存模板说明。
2. 存适用场景。
3. 存关键词。
4. 存示例 plan。
5. 存是否已验证。
6. 帮 AI 选择 template_id。

代码节点职责：
1. 根据 template_id 调用白名单模板函数。
2. 生成真正可导入的 nodes/edges/chatConfig。
3. 修正 output id 引用。
4. 遇到未支持模板，返回提示型工作流，不乱造。

不要让知识库返回完整巨大 JSON 的原因：
1. 知识库切片可能把 JSON 切断。
2. 检索可能只返回一半。
3. AI 可能拼错。
4. JSON 结构变动后不好维护。
5. 已验证模板应由代码节点固定生成。

二十八、常见错误与最终修正
======================================================================

错误 1：
AI 节点 401 Invalid token。

旧判断：
可能平台 AI 节点不能用。

最新修正：
云版 4.15.0 后续测试 AI 可用。若再出现 401，优先检查具体模型配置，不要直接否定整个生成器。

----------------------------------------------------------------------

错误 2：
AI 直接生成完整 JSON。

问题：
容易生成 sourceNodeId/targetNodeId 这种不匹配真实导出的结构。

修正：
AI 只生成 plan，代码节点生成 JSON。

----------------------------------------------------------------------

错误 3：
测试 A/B 报 Data validation error。

问题：
AI → 代码清洗 → 指定回复链路设计不稳；AI 输出结构和校验代码不匹配；后续又发现代码节点输出引用 key/id 问题。

修正：
先 D1/D2 验证 AI→代码，再使用固定模板。

----------------------------------------------------------------------

错误 4：
随机数生成器生成了“流程开始 → 指定回复：该需求需要代码节点”。

原因：
V2 为了避免代码节点错误，先把随机数识别为 need_code 并拦截。

修正：
V3 加入 code_task 模板。

----------------------------------------------------------------------

错误 5：
代码节点 user_input 输入框显示 ["448745","userChatInput"]，并且左边显示手动输入。

原因：
自定义输入变量引用格式不对，被当成普通字符串。

修正：
无输入 code_task 先删除 user_input。
带输入 code_task 以后单独导出母版再加。

----------------------------------------------------------------------

错误 6：
代码节点 outputs 里写了 result，但 UI 没显示 result。

原因：
自动生成结构不完全符合云版 4.15.0，特别是输出 id/key 和 answerNode 引用关系。

修正：
用真实导出母版后确认：answerNode 要引用 output.id。

----------------------------------------------------------------------

错误 7：
最终输出中同时出现 plan JSON 和 workflow JSON。

原因：
生成器里的 AI 需求理解器开启了“直接回复用户”。

修正：
生成器里的 AI 节点必须关闭直接回复用户。
最终指定回复只引用代码模板生成器.result。

----------------------------------------------------------------------

错误 8：
导入时报错，可能是复制了两个 JSON。

原因：
用户把 plan JSON 和 workflow JSON 一起复制导入。

修正：
导入时只能复制第二段：
{
  "nodes": [...],
  "edges": [...],
  "chatConfig": {...}
}

二十九、最短给下一个对话的提醒
======================================================================

如果下一个对话继续做这个项目，先看这里：

我们在 FastGPT 云版 4.15.0 做“一句话生成工作流 JSON”的生成器。现在不是旧阶段了，AI 节点已经能用，simple_reply、ai_chat、code_task.random_number 都已经实测成功。真正稳定架构是：workflowStart → AI 需求理解器 → 代码模板生成器 → answerNode 输出 JSON。AI 只生成 plan，不直接生成 nodes/edges。代码节点根据 plan.workflow_type 调用模板函数生成 JSON。

最关键修正：代码节点 result 输出被 answerNode 引用时，引用的是 output.id，不是 key。成功导出的代码节点 outputs 里 result 是：
{id:"qLUQfhG0ILRX", key:"result", valueType:"string"}
answerNode 引用：
["x2SKHIaS2Ra6SHGq","qLUQfhG0ILRX"]
所以生成器里固定用：
代码节点输出 id = "code_result_output"
key = "result"
answerNode.value = ["code_task_node","code_result_output"]

不要再写 ["code_task_node","result"]，那会导致 Data validation error。

当前已成功模板：
1. simple_reply.basic.v1：流程开始 → 指定回复。
2. ai_chat.basic.v1：流程开始 → AI 对话。
3. code_task.random_number.v1：流程开始 → 代码运行 → 指定回复。

下一步如果要扩展，要么添加无输入 code_task（时间、验证码、UUID、密码、固定计算），要么先手动搭一个“带 user_input 输入变量的代码节点母版”并导出，再支持文本处理/动态计算。文件、判断器、知识库、HTTP、数据库都要先导出母版，不能让 AI 直接乱写。

附录A：最新代码模板生成器 JS【当前推荐版本】

function main(params) {
  const ai_text = params && params.ai_text !== undefined ? String(params.ai_text) : '';

  function cleanJsonText(text) {
    let clean = String(text || '').trim();

    clean = clean
      .replace(/^```json\s*/i, '')
      .replace(/^```\s*/i, '')
      .replace(/```$/i, '')
      .trim();

    const start = clean.indexOf('{');
    const end = clean.lastIndexOf('}');
    if (start >= 0 && end > start) {
      clean = clean.slice(start, end + 1);
    }

    return clean;
  }

  function safeText(value, fallback) {
    const text = value === undefined || value === null ? '' : String(value);
    return text.trim() ? text : fallback;
  }

  function makeChatConfig() {
    return {
      variables: [],
      scheduledTriggerConfig: {
        cronString: '',
        timezone: 'Etc/GMT-8',
        defaultPrompt: ''
      }
    };
  }

  function makeUserGuide(welcomeText) {
    return {

  function normalizeType(type) {
    const t = String(type || '').trim().toLowerCase();

    if (t === 'reply' || t === 'fixed_reply') return 'simple_reply';
    if (t === 'chat' || t === 'assistant' || t === 'llm_chat' || t === 'ai') return 'ai_chat';

    if (
      t === 'random_number' ||
      t === 'random' ||
      t === 'calculate' ||
      t === 'calculator' ||
      t === 'text_process' ||
      t === 'current_time'
    ) {
      return 'code_task';
    }

    if (t === 'file_parse' || t === 'readfiles' || t === 'image_ocr') return 'file_task';

    return t;
  }

  function defaultCodeByTask(task, expression) {
    const t = String(task || '').toLowerCase();

    if (t === 'random_number' || t === 'random') {
      return [
        'function main(params) {',
        '  const n = Math.floor(Math.random() * 1000000);',
        '  return { result: String(n) };',
        '}'
      ].join('\n');
    }

    if (t === 'current_time' || t === 'time') {
      return [
        'function main(params) {',
        '  const now = new Date();',
        '  return { result: now.toLocaleString("zh-CN", { timeZone: "Asia/Shanghai" }) };',
        '}'
      ].join('\n');
    }

    if (t === 'calculator' || t === 'calculate') {
      const expr = String(expression || '1+1').replace(/[^0-9+\-*/().%\s]/g, '');
      return [
        'function main(params) {',
        '  const expression = ' + JSON.stringify(expr) + ';',
        '  let value;',
        '  try {',
        '    value = Function("return (" + expression + ")")();',
        '  } catch (err) {',
        '    value = "计算失败";',
        '  }',
        '  return { result: String(value) };',
        '}'
      ].join('\n');
    }

    if (t === 'text_length') {
      return [
        'function main(params) {',
        '  const input = String(params.user_input || "");',
        '  return { result: String(input.length) };',
        '}'
      ].join('\n');
    }

    return [
      'function main(params) {',
      '  return { result: "代码任务已运行" };',
      '}'
    ].join('\n');
  }

  function ensureSafeJsCode(code, plan) {
    let js = String(code || '').trim();

    if (!js || js.indexOf('function main') === -1) {
      js = defaultCodeByTask(plan.task, plan.expression);
    }

    if (js.indexOf('return') === -1 || js.indexOf('result') === -1) {
      js = defaultCodeByTask(plan.task, plan.expression);
    }

    return js;
  }

  let plan = {};
  try {
    plan = JSON.parse(cleanJsonText(ai_text));
  } catch (err) {
    return {
      result: JSON.stringify(
        makeSimpleReplyWorkflow('AI 输出不是合法 JSON，生成失败。请检查 AI 节点是否只输出 plan JSON。'),
        null,
        2
      )
    };
  }

  const workflowType = normalizeType(plan.workflow_type || plan.type);
  let workflow;

  if (workflowType === 'simple_reply') {
    const replyText = safeText(plan.reply_text || plan.text || plan.message, 'hello');
    workflow = makeSimpleReplyWorkflow(replyText);
  } else if (workflowType === 'ai_chat') {
    const systemPrompt = safeText(
      plan.system_prompt || plan.prompt,
      '你是一个友好、简洁的 AI 助手。请根据用户问题直接回答。'
    );
    const modelName = safeText(plan.model, 'Qwen-flash');
    workflow = makeAiChatWorkflow(systemPrompt, modelName);
  } else if (workflowType === 'code_task') {
    const nodeName = safeText(plan.name || plan.task_name, '代码运行');
    const jsCode = ensureSafeJsCode(plan.js_code || plan.code, plan);
    workflow = makeCodeWorkflow(jsCode, nodeName);
  } else if (workflowType === 'file_task') {
    const reason = safeText(
      plan.reason,
      '该需求需要文件上传、readFiles 或 AI 视觉节点。当前版本暂不自动生成文件处理流程。'
    );
    workflow = makeSimpleReplyWorkflow(reason);
  } else if (workflowType === 'unsupported') {
    const reason = safeText(
      plan.reason,
      '该需求需要数据库、HTTP、知识库或工具调用等外部配置，不能直接写死。'
    );
    workflow = makeSimpleReplyWorkflow(reason);
  } else {
    workflow = makeSimpleReplyWorkflow(
      '未识别 workflow_type：' +
      String(plan.workflow_type || plan.type || '') +
      '。当前支持 simple_reply、ai_chat、code_task。'
    );
  }

  return {
    result: JSON.stringify(workflow, null, 2)
  };
}

一、当前已经确认的核心事实
======================================================================

1. 当前方向已经从“单个 hello 流程测试”升级为：
   一句话生成 FastGPT 工作流 JSON 的模板库生成器。

2. 已验证的主链路是：
   用户一句话
   ↓
   AI 需求理解器输出 plan JSON
   ↓
   代码模板生成器根据 plan 选择模板并拼真实 FastGPT JSON
   ↓
   指定回复输出 JSON
   ↓
   用户复制导入 FastGPT
   ↓
   生成的新工作流运行

3. 不能再让 AI 直接自由生成完整 nodes/edges/chatConfig。
   原因：
   - AI 会猜错 edges 字段，例如 sourceNodeId/targetNodeId。
   - AI 可能引用不存在的输出。
   - AI 可能把 key 当成输出引用，而云版 4.15.0 实际需要引用 output.id。
   - AI 可能生成 datasetId、toolId、数据库连接、HTTP 配置等不能写死的内容。

4. 正确分工：
   - AI：理解需求，输出 plan。
   - 知识库：检索该用哪个模板。
   - 代码节点：按已验证母版生成真实 JSON。
   - 校验逻辑：兜底、提示、避免乱生成。
   - 用户：复制 JSON 导入并测试。

5. 当前已经验证成功的模板：
   - simple_reply：流程开始 → 指定回复。
   - ai_chat：流程开始 → AI 对话。
   - code_task.no_input：流程开始 → 代码运行 → 指定回复；随机数已成功。

6. 随机数模板最关键修正：
   answerNode 引用代码节点 result 时，不能写：
   ["code_task_node", "result"]

   必须引用代码节点 outputs 里的 id，例如：
   ["code_task_node", "code_result_output"]

   并且代码节点 outputs 必须包含：
   {
     "id": "code_result_output",
     "key": "result",
     "valueType": "string",
     "label": "result"
   }

7. 代码节点真正成功母版导出的结构里：
   - code 节点有 system_rawResponse。
   - code 节点有 error。
   - 自定义 result 输出有独立 id。
   - 指定回复引用 output.id，而不是 key。
   - chatConfig 里可有 variables 和 scheduledTriggerConfig，但不要写死 _id。

======================================================================
六、知识库模板卡片格式
======================================================================

建议建立知识库名称：
FastGPT工作流模板库

每个模板一张卡片，不要直接放完整巨大 JSON。
模板卡片格式：

TEMPLATE_ID:
WORKFLOW_TYPE:
STATUS:
FASTGPT_VERSION:
STRUCTURE:
KEYWORDS:
WHEN_TO_USE:
WHEN_NOT_TO_USE:
REQUIRED_PARAMS:
OUTPUT_RULES:
KNOWN_PITFALLS:
EXAMPLE_USER_REQUEST:
EXAMPLE_PLAN:
NOTES:

示例卡片：

TEMPLATE_ID: code_task.no_input.v1
WORKFLOW_TYPE: code_task
STATUS: tested_cloud_4.15.0
FASTGPT_VERSION: cloud_4.15.0
STRUCTURE: workflowStart -> code -> answerNode
KEYWORDS: 随机数, 验证码, 当前时间, UUID, 密码, 固定表达式计算
WHEN_TO_USE: 不需要用户运行时输入，代码自己生成结果
WHEN_NOT_TO_USE: 需要读取用户输入、文件、图片、知识库、数据库
REQUIRED_PARAMS: name, js_code
OUTPUT_RULES:
- JS 必须 function main(params)
- return { result: String(...) }
- code output id 必须是 code_result_output
- answerNode 引用 ["code_task_node", "code_result_output"]
KNOWN_PITFALLS:
- 不要引用 ["code_task_node","result"]
- 不要把 output key 当成引用 id
EXAMPLE_USER_REQUEST: 生成一个随机数
EXAMPLE_PLAN:
{
  "template_id": "code_task.no_input.v1",
  "workflow_type": "code_task",
  "task": "random_number",
  "name": "生成随机数",
  "js_code": "function main(params) { const n = Math.floor(Math.random() * 1000000); return { result: String(n) }; }"
}

七、知识库驱动生成器目标流程
======================================================================

当前 V3：
流程开始
↓
AI 需求理解器
↓
代码模板生成器
↓
指定回复输出 JSON

未来 V4：
流程开始
↓
AI 生成检索 query
↓
知识库搜索“FastGPT工作流模板库”
↓
AI 根据检索结果选择 template_id，并补全 params
↓
代码模板生成器根据 template_id 生成 JSON
↓
指定回复输出 JSON

V4 的关键：
1. 知识库只负责检索模板说明，不直接返回完整 JSON。
2. AI 只能选择知识库里存在的 template_id。
3. 代码节点只允许白名单 template_id。
4. 未验证模板必须生成提示型工作流，不能硬生成。
5. datasetId 不能写死，知识库节点要在生成器应用里手动配置。

3. 导出 JSON。

5. 找到关键输出 id，而不是只看 key。
6. 找到 answerNode 的真实引用方式。
7. 找到 edges 的 source/target/sourceHandle/targetHandle。
8. 写成模板函数。
9. 让一句话生成器生成同类 JSON。
10. 新建应用导入生成 JSON。
11. 运行成功。
12. 加入模板知识库，状态标记 tested_cloud_4.15.0。

如果没有第 1 到 3 步，不能把模板标成已验证。

answerNode 引用动态输出时优先引用 output.id，不要只引用 key。
