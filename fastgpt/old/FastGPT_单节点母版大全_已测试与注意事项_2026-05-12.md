## 1. 全局生成规则

### 1.1 引用格式有两种

#### 输入框 value 里引用节点输出

多数节点输入参数里，变量引用是数组：

```json
["节点ID", "输出ID"]
```

例如：

```json
"value": ["448745", "userChatInput"]
```

#### 文本框里插入变量

指定回复、说明文字等文本内容里，变量引用是：

```text
{{$节点ID.输出ID$}}
```

例如：

```text
{{$gI9sxln4G2T2uQb3.formInputResult$}}
```

### 1.2 全局变量特殊节点 ID

全局变量使用特殊节点 ID：

```text
VARIABLE_NODE_ID
```

例如：

```text
{{$VARIABLE_NODE_ID.userId$}}
{{$VARIABLE_NODE_ID.appId$}}
{{$VARIABLE_NODE_ID.cTime$}}
```

### 1.3 下游引用必须看 outputs.id

不能只看 `key`、`label` 或界面显示名。尤其表单字段修改名称后，可能出现：

```json
{
  "id": "文本",
  "key": "时间分钟",
  "label": "时间分钟"
}
```

这种情况下下游文本引用仍然要用：

```text
{{$表单节点ID.文本$}}
```

而不是自动猜：

```text
{{$表单节点ID.时间分钟$}}
```

### 1.4 常见边 handle

| 场景 | sourceHandle |
|---|---|
| 普通成功出口 | `节点ID-source-right` |
| 普通入口 | `节点ID-target-left` |
| 错误捕获出口 | `节点ID-source_catch-right` |
| 工具调用挂载 | `sourceHandle: selectedTools`, `targetHandle: selectedTools` |
| 用户选择某选项 | `用户选择节点ID-source-选项key` |
| 判断器 IF | `判断器节点ID-source-IF` |
| 判断器 ELSE IF 1 | `判断器节点ID-source-ELSE IF 1` |
| 判断器 ELSE | `判断器节点ID-source-ELSE` |

## 2. 系统配置 userGuide

`userGuide` 是系统配置节点，一般不参与业务连线。

核心字段包括：

- `welcomeText`
- `variables`
- `questionGuide`
- `tts`
- `whisper`
- `scheduleTrigger`

其中自动执行不在 `userGuide` 内，而在 `chatConfig.autoExecute`。

## 3. 流程开始 workflowStart

普通文本输入版：

```json
{
  "flowNodeType": "workflowStart",
  "inputs": [
    {
      "key": "userChatInput",
      "renderTypeList": ["reference", "textarea"],
      "valueType": "string",
      "required": true
    }
  ],
  "outputs": [
    {
      "id": "userChatInput",
      "key": "userChatInput",
      "valueType": "string",
      "type": "static"
    }
  ]
}
```

注意：如果某些节点引用了：

```json
["流程开始节点ID", "userFiles"]
```

则流程开始必须启用文件上传并具备 `userFiles` 输出。否则就是悬空引用。本轮工具调用测试文件里出现过 `fileUrlList -> userFiles`，但对应 `workflowStart` 没有 `userFiles` 输出，这属于测试文件里的潜在冲突，最终模板要修正。

  "valueType": "any",
  "required": true

```text
{{$节点ID.answerText$}}
{{$节点ID.result$}}

```

但必须以目标节点真实 `outputs.id` 为准。

## 5. 数据库连接 tool / systemTool-databaseConnection

数据库连接不是 `databaseNode`，而是系统工具节点。

核心：

```json
{
  "flowNodeType": "tool",
  "pluginId": "systemTool-databaseConnection",
  "toolConfig": {
    "systemTool": {
      "toolId": "systemTool-databaseConnection"
    }
  }
}
```

输入分两部分：

### 5.1 数据库连接配置

在 `system_input_config.inputList` 中：

- `databaseType`：MySQL / PostgreSQL / Microsoft SQL Server
- `host`
- `port`
- `databaseName`
- `user`
- `password`

这些都不能写死真实配置。

### 5.2 SQL 输入

```json
{
  "key": "sql",
  "renderTypeList": ["input", "reference"],
  "valueType": "string",
  "required": true
}
```

输出：

| id | valueType | 说明 |
|---|---|---|
| `result` | object | SQL 执行结果 |
| `system_error_text` | string | 错误信息 |

推荐结构：

```text
流程开始 → 代码运行生成 SQL → 数据库连接执行 SQL → 指定回复输出 result
```

不要继续用 HTTP 请求伪装数据库模块。HTTP 只是代理入库方案，不是数据库模块本身。

## 6. 工具调用 tools

工具调用节点：

```json
{
  "flowNodeType": "tools",
  "avatar": "core/workflow/template/toolCall",
  "version": "4.9.2"
}
```

输出：

- `answerText`
- `system_error_text`

工具挂载必须使用特殊边：

```json
{
  "sourceHandle": "selectedTools",
  "targetHandle": "selectedTools"
}
```

典型结构：

```text
流程开始 → 工具调用 → 指定回复
工具调用 --selectedTools--> AI 对话 / 知识库搜索 / 其他可作为工具的节点
工具内部节点 → 工具调用终止 stopTool
```

注意：工具调用测试只能证明“能挂载”，不等于被挂载节点作为普通主流程已经测试成功。普通主流程还要单独导出测试。

## 7. 工具调用终止 stopTool

结构：

```json
{
  "flowNodeType": "stopTool",
  "avatar": "core/workflow/template/stopTool",
  "inputs": [],
  "outputs": []
}
```

作用：工具内部流程执行到这里时，终止本次工具调用，不再让 AI 根据工具结果继续回答。

## 9. 用户选择 userSelect

正式主流程测试结构：

```text
流程开始 → 用户选择 → 多个指定回复分支
```

核心：

```json
{
  "flowNodeType": "userSelect",
  "avatar": "core/workflow/template/userSelect",
  "inputs": ["description", "userSelectOptions"],
  "outputs": ["selectResult"]
}
```

`userSelectOptions` 示例：

```json
[
  {"value": "这是选项1", "key": "option1"},
  {"value": "这是选项2", "key": "option2"},
  {"value": "这是选项3", "key": "jNIKegS0jwQnUKXR"}
]
```

分支边规则：

```text
sourceHandle = 用户选择节点ID + "-source-" + 选项key
```

例：

```json
"sourceHandle": "hjyZ5IKUflnaARZS-source-option1"
```

输出只有：

```text
selectResult
```

不要编 `choice`、`selected`、`optionValue`。

## 10. 表单输入 formInput

正式主流程测试结构：

```text
流程开始 → 表单输入 → 指定回复
```

核心：

```json
{
  "flowNodeType": "formInput",
  "avatar": "core/workflow/template/formInput",
  "inputs": ["description", "userInputForms"],
  "outputs": ["formInputResult", "每个字段的单独输出"]
}
```

### 10.1 表单说明支持变量

`description` 可写：

```text
{{$448745.userChatInput$}}
```

### 10.2 每个表单字段都会生成一个输出

- `formInputResult`：完整对象
- 每个字段 key/id：单独输出

引用方式：

```text
{{$表单节点ID.formInputResult$}}
{{$表单节点ID.字段输出ID$}}
```

### 10.3 已确认 10 种字段类型

| 界面名称 | type | valueType | 关键字段 |
|---|---|---|---|
| 文本输入框 | `input` | `string` | `defaultValue`, `maxLength` |
| 密码 | `password` | `string` | `defaultValue`, `minLength` |
| 数字输入框 | `numberInput` | `number` | `defaultValue`, `min`, `max` |
| 单选框 | `select` | `string` | `list`, `defaultValue` |
| 多选框 | `multipleSelect` | `arrayString` | `list`, `defaultValue` 数组 |
| 开关 | `switch` | `boolean` | `defaultValue` true/false |
| 时间点 | `timePointSelect` | `string` | `timeGranularity`, `timeRangeStart`, `timeRangeEnd` |
| 时间范围 | `timeRangeSelect` | `arrayString` | `defaultValue` 开始/结束数组 |
| 文件上传 | `fileSelect` | `arrayString` | 上传开关、`maxFiles` |
| 对话模型选择 | `selectLLMModel` | `string` | `defaultValue` 模型名 |

### 10.4 必填开关

界面“必填”开关对应：

```json
"required": true
```

或：

```json
"required": false
```

### 10.5 时间粒度

目前确认：

| 界面 | JSON |
|---|---|
| 小时 | `hour` |
| 分钟 | `minute` |
| 秒 | `second` |

时间格式统一建议：

```text
YYYY-MM-DD HH:mm:ss
```

即使粒度是小时，也写成：

```text
2026-05-03 02:00:00
```

### 10.6 文件上传字段

结构：

```json
{
  "type": "fileSelect",
  "valueType": "arrayString",
  "canLocalUpload": true,
  "canUrlUpload": true,
  "canSelectFile": true,
  "canSelectImg": true,
  "canSelectVideo": true,
  "canSelectAudio": true,
  "maxFiles": 20
}
```

输出是一组文件链接，可接文档解析、AI 对话 fileUrlList、代码节点后缀分析。

### 10.7 冲突/坑

- 字段后期改名时，`outputs.id` 可能残留旧值。下游引用以 `outputs.id` 为准。
- 单选/多选的 `value` 不建议重复。
- 生成模板时建议用英文或拼音 key：`text_input`、`amount`、`upload_files`，减少引用错误。

## 11. 变量更新 variableUpdate

正式主流程测试结构：

```text
流程开始 → 变量更新 → 指定回复
```

核心：

```json
{
  "flowNodeType": "variableUpdate",
  "avatar": "core/workflow/template/variableUpdate",
  "showStatus": false,
  "outputs": []
}
```

核心输入：

```json
{
  "key": "updateList",
  "valueType": "any",
  "renderTypeList": ["hidden"]
}
```

更新项结构：

```json
{
  "variable": ["目标节点ID", "目标输出ID或变量名"],
  "value": ["来源节点ID", "来源输出ID或固定值"],
  "valueType": "string",
  "renderType": "reference"
}
```

固定值结构：

```json
{
  "variable": ["VARIABLE_NODE_ID", "appId"],
  "value": ["", "这是值"],
  "valueType": "string",
  "renderType": "input"
}
```

规则：

- `variableUpdate` 没有 outputs，不能被下游引用结果。
- 更新全局变量时，目标节点 ID 是 `VARIABLE_NODE_ID`。
- `renderType: reference` 表示值来自引用。
- `renderType: input` 表示值是固定输入。
- 测试文件中第二个分支更新 `appId`，但回复引用了 `userId`，验证时应改为引用 `appId`。

## 12. 批量执行 loop / loopStart / loopEnd

批量执行不是单节点，它是一个容器。

完整结构：

```text
loop 父节点
├─ loopStart 子节点
├─ 中间处理节点，如判断器、代码、AI、指定回复等
└─ loopEnd 子节点
```

### 12.1 loop 父节点

核心：

```json
{
  "flowNodeType": "loop",
  "avatar": "core/workflow/template/loop"
}
```

关键输入：

- `loopInputArray`
- `childrenNodeIdList`
- `nodeWidth`
- `nodeHeight`
- `loopNodeInputHeight`

输出：

```text
loopArray
```

### 12.2 childrenNodeIdList 与 parentNodeId 必须同时存在

`loop.childrenNodeIdList` 必须包含循环内部节点 ID。

循环内部节点必须带：

```json
"parentNodeId": "loop节点ID"
```

两者缺一不可。

### 12.3 loopStart 输出

```text
loopStartIndex：当前数组元素下标
loopStartInput：当前数组元素
```

循环体内处理当前元素时，应引用：

```text
{{$loopStart节点ID.loopStartInput$}}
{{$loopStart节点ID.loopStartIndex$}}
```

### 12.4 loopEnd 输入

`loopEndInput` 是每次循环要收集进结果数组的值。

父节点最终输出：

```text
{{$loop节点ID.loopArray$}}
```

### 12.5 建议模板

不要直接让 loop 引用普通字符串，最好先用代码节点输出数组：

```text
流程开始 → 代码运行拆分为 arrayString → 批量执行 → 指定回复输出 loopArray
```

## 17. 应用调用 appModule

Agent 页里拖入已有应用，会生成 `appModule` 节点。

核心：

```json
{
  "flowNodeType": "appModule",
  "avatar": "core/app/type/workflowFill",
  "pluginId": "被调用应用ID"
}
```

注意：这里的 `pluginId` 是被调用应用的 ID，不能编造。

输入：

- `system_forbid_stream`：禁用流输出
- `history`
- `userChatInput`

输出：

| id | 说明 |
|---|---|
| `history` | 新上下文 |
| `answerText` | 被调用应用回复内容 |
| `system_error_text` | 错误信息 |

指定回复通常引用：

```text
{{$appModule节点ID.answerText$}}
```

注意：

- `system_forbid_stream = true`：嵌套应用非流式运行。
- `catchError` 可为 true/false，但 appModule 的错误分支 handle 本轮未单独测试。
- 生成 `appModule` 时必须先有用户真实导出的应用节点，不能凭空写 `pluginId`。

---

## 18. 自动执行 chatConfig.autoExecute

自动执行不是节点，在 `chatConfig` 里。

打开：

```json
"autoExecute": {
  "open": true,
  "defaultPrompt": ""
}
```

关闭：

```json
"autoExecute": {
  "open": false,
  "defaultPrompt": ""
}
```

不要把它写进 `nodes`。

## 19. 冲突与自检清单

生成工作流 JSON 后必须检查：

1. 所有 `value: [nodeId, outputId]` 的 nodeId 存在。
2. 所有 `outputId` 必须在该节点 `outputs.id` 里存在。
3. 文本变量 `{{$nodeId.outputId$}}` 也必须检查 outputs.id。
4. 如果引用 `userFiles`，`workflowStart` 必须真的有 `userFiles` 输出。
5. `userSelect` 的每个选项分支 handle 必须是 `用户选择节点ID-source-选项key`。
6. `loop` 内部节点必须同时满足：`parentNodeId = loop节点ID`，并且在 `childrenNodeIdList` 中。
7. `variableUpdate` 没有输出，不要让下游引用它。
8. `catchError: true` 的错误分支使用 `节点ID-source_catch-right`。
9. 企业微信、钉钉 webhook 没有正常 result 输出；飞书 webhook 有 `result`。
10. 数据库连接、webhook、appModule 的真实私密配置不能写死。
11. appModule 的 `pluginId` 必须来自真实应用，不能编造。
12. 表单字段改名后检查 `outputs.id/key/label` 是否一致；引用以 `outputs.id` 为准。
13. 单选/多选的 value 不要重复。
14. 时间字段统一写完整 `YYYY-MM-DD HH:mm:ss`。
15. 模型名、知识库 ID、数据库连接、webhook 地址、secret、appModule pluginId 都是环境相关字段，除非来自用户导出文件，否则不能硬写。
